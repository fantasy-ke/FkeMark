## 13. 修复长文档连续输入阻塞

### 需求
- 修复超长 Markdown 在实时编辑模式下每次输入都长时间卡住、连续输入无法正常进行的问题。
- 保留此前对长文档首次加载以及实时编辑、分栏、阅读模式切换的性能优化。

### 根因
- TipTap 的 `onUpdate` 在每个文档事务中同步执行 `editor.getHTML()` 和 `htmlToMarkdown()`，输入一个字符就会遍历并转换整篇长文档，主线程因此在每次按键后被阻塞。

### 实现
- 新增 `useDeferredEditorChange`：超过现有 100000 字符阈值的文档在实时编辑时立即标记为已修改，但将整篇 HTML → Markdown 序列化延迟到停止输入 300ms 后执行；连续输入只转换一次。
- 短文档继续保持原有即时同步行为，不改变普通文档的编辑反馈。
- `EditorHandle.getContent()` 可主动刷新并取消待处理序列化，保存、导出、模式切换、标签切换/关闭以及安装更新前保存都会读取最新编辑器内容，避免延迟同步造成内容丢失。
- 将当前编辑器内容读取与模式切换协调拆到 `useCurrentEditorContent`，使 `App.tsx` 保持低于 800 行。
- 增加性能回归测试，覆盖连续输入不立即转换整篇文档、停顿后只转换一次，以及立即保存时刷新待处理内容。

## 14. 修复未保存双向链接不显示反向引用

### 需求
- 修复源笔记已成功插入双向链接，但切换到目标笔记后，反向链接面板没有显示来源引用的问题。
- 未保存内容必须能够参与反向链接扫描，同时保留未打开文件从磁盘读取的现有行为。

### 根因
- 反向链接面板只通过 `read_file_command` 或浏览器读取接口扫描磁盘内容。
- 用户创建双向链接后切换到目标笔记时，源笔记的最新内容已保存在标签页内存缓存中，但磁盘文件可能尚未保存，因此扫描到的是旧内容。

### 实现
- 将现有 `tabContentCache` 传入反向链接面板。
- 扫描已打开的 Markdown 文件时，按规范化后的文件路径优先使用标签缓存内容；缓存中不存在的文件继续读取磁盘。
- 不增加自动保存或额外索引层，避免改变现有保存语义和扫描流程。
- 增加组件回归测试，确认磁盘内容尚未包含链接时，面板仍可从未保存标签内容中显示反向引用。
