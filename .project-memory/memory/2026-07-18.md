4. **SHA256SUMS 资产校验**：GoNavi release 阶段对全部产物生成 `SHA256SUMS` 供用户校验完整性。FkeMark 当前未生成，发布正式版时可作为信任增强项。
5. **静态更新清单**：GoNavi 生成 `latest.json` / `latest-dev.json` 静态清单放到 release assets，客户端通过 `github.com/.../latest/download/latest.json` 检查更新，避免打爆 api.github.com 未认证配额。FkeMark 当前 `includeUpdaterJson: false`（未启用 updater），若未来开启 Tauri updater 可参考此模式。
6. **changelog 分类**：GoNavi release.yml 用 emoji + 关键字双匹配（✨/feat、🐛/fix、⚡/perf、♻️/refactor、🌐、🔧/chore）分类提交记录，并附 `compare/prevTag...curTag` 链接。FkeMark 当前用 Python + `.github/changelog.yml` 的 prefixes 分类（更严格、不依赖 emoji），已能满足需求，但**可借鉴 compare 链接**增强正式版 Release 正文。
7. **MSI 版本号映射**：GoNavi 从 git tag `vX.Y.Z` 解析 major/minor/patch 映射到 WiX MSI productVersion（major.minor.build，build≤65535）。FkeMark 当前由 tauri-action 自动处理 msi 版本，若需自定义可参考。
8. **Linux AppImage 工具链**：GoNavi 显式下载 `linuxdeploy` + `linuxdeploy-plugin-gtk` 并设置 `DEPLOY_GTK_VERSION=3`，arm64 跳过 AppImage 仅产出 tar.gz。FkeMark 由 tauri-action 内部处理 appimage，若遇 AppImage 失败可参考其兜底逻辑（`touch /tmp/skip-appimage` + `exit 0`）。

## 流水线重构（借鉴 GoNavi + 适配 Tauri v1）

### 需求
用户要求：读取 GoNavi 流水线文件，调整 FkeMark 的流水线文件，使打包产物和流程对齐 GoNavi 风格；dev 版本号采用 `dev-d0a6a97`（短 SHA7）格式；msi 打包不能出错。

### msi 打包不出错的关键决策
**根因**：Tauri v1 的 msi 用 WiX 打包，版本号必须是 `major.minor.patch` 纯数字格式。若把 `dev-d0a6a97` 这种带字母/连字符的版本号写入 `tauri.conf.json` 或 `Cargo.toml`，WiX 会报错导致 msi 打包失败。

**解决方案**（借鉴 GoNavi MSI 版本号映射思路）：
- **dev 构建**：**不修改** `Cargo.toml`/`tauri.conf.json` 的版本号（保持 `0.1.0`），msi 用 `0.1.0` 合法版本号打包。`dev-d0a6a97` 仅作为 Release name + 产物文件名后缀（显示用），不传入 Tauri 构建链路。
- **release 构建**：从 tag `vX.Y.Z` 提取数字版本号 `X.Y.Z`，构建前用 Python 写入 `Cargo.toml`/`tauri.conf.json`/`package.json`，msi 用合法 `X.Y.Z` 打包。Python 脚本含正则校验 `^\d+\.\d+\.\d+$`，非法版本号直接 fail。

## 二次重做（分支切换后，融入更多 GoNavi 特殊设置）

### 背景
用户首次调整时弄错了分支（改动落在错误分支），随后切换到 `dev` 分支并 revert 了之前的尝试 commit（`94a6131 Revert "feat(ci): ..."`)。git log 显示之前还试过用 `0.1.0-<数字>` 这种 semver 预发布格式（`ea224b5`）让 Tauri 打包，但 Tauri v1 msi/WiX 对 semver 预发布标识符支持有问题而失败——验证了「dev 保持 0.1.0 不改、release 写入纯数字版本号」方案的正确性。

### 本次新增的 GoNavi 特殊设置（在首次重构基础上补强）
1. **顶层 `permissions: contents: write`**：从每个 job 单独声明改为 workflow 顶层全局声明，简化配置。
2. **`actions/github-script@v7` 重置 dev-latest**（替换原 gh CLI）：借鉴 GoNavi，用 `github.rest.repos.listReleases` 分页查找 + 逐个 `deleteRelease` + `git.deleteRef`，并对 404/422 容错。比 `gh release delete || true` 更可靠（gh CLI 有时分页不全/认证问题）。
3. **`target_commitish: ${{ github.sha }}`**：dev release 指向具体 commit，避免 release 指向错误 ref。
4. **`Format build time` 步骤**：用 `TZ='Asia/Shanghai' date '+%Y-%m-%d %H:%M:%S'` 生成北京时间构建时间（比解析 `head_commit.timestamp` 更可靠，避免空值导致整个发布失败）。首次尝试用 Python 解析 `head_commit.timestamp`，已改为 date 命令。
5. **release body 含元数据表格**：dev release body 追加版本/分支/提交链接/构建时间的 Markdown 表格，方便测试者识别。
6. **`if-no-files-found: error`**：所有 upload-artifact 步骤启用严格模式，产物缺失直接报错而非静默成功。
7. **`retention-days: 7`**：dev/release 构建 artifact 保留 7 天。
8. **`make_latest: true`**（release.yml）：create-release 与 publish 上传步骤均设置，标记为最新 release。
9. **changelog compare 链接**（release.yml）：changelog 末尾追加 `[v0.1.0...v0.2.0](.../compare/...)`。
10. **`merge-multiple: true`**（download-artifact）：所有平台产物合并到同一目录，便于统一生成 SHA256SUMS。

### 教训：IDE 编辑器缓冲区会覆盖外部写入
- 首次 `write` dev.yml 报告成功，但随后 `git diff` 显示 dev.yml 无变化——磁盘文件被 IDE 编辑器缓冲区还原成旧版本（用户在 IDE 中打开了 dev.yml，cursor 在 line 22）。
- release.yml 因未在 IDE 打开，写入正常保留。
- **对策**：写入后必须立即用 `git diff --stat` 验证磁盘真实内容，不能只信 write 工具的返回。若发现被还原，重新 write 并再次验证。
- 后续编辑 workflow 文件时，若用户在 IDE 打开了该文件，写入后应提示用户重新加载。

## 修复 dev 构建版本号显示与检查更新失败（第三次调整）

### 用户反馈的 3 个问题（本机构建后实测）
1. dev 构建安装后，本机构建显示在"正式版"中（通道徽章位置错）
2. 开发版显示的版本号是 `v0.1.0`，需要显示 `vdev-**`（带 v 前缀）
3. 检查更新会失败报错

### 根因分析
- 问题1：CI 未设 `VITE_UPDATE_CHANNEL=dev`，vite 注入默认 `'latest'`，`getBuildChannel()` 返回 latest
- 问题2：`vite.config.ts` 的 `__APP_VERSION__` 取自 `pkg.version`（0.1.0），CI 未注入 `VITE_APP_VERSION=dev-<SHA>`
- 问题3：CI 未生成 `latest.json`/`dev.json` 静态清单（404），降级 GitHub API 时通道判断错

### 关键设计：分离"打包版本号"与"显示版本号"
- 打包版本号（msi/WiX 用，纯数字）：dev 保持 `0.1.0`，release 写入 `X.Y.Z`
- 显示版本号（前端 UI 用，可含 dev-）：通过 Vite 环境变量 `VITE_APP_VERSION` 注入 `__APP_VERSION__`

### 修改的文件
1. `vite.config.ts`：`__APP_VERSION__` 改为 `process.env.VITE_APP_VERSION || pkg.version`
2. `src/components/AboutPage.tsx`：加 `appVersion` prop，3 处硬编码改为 `v{appVersion || '0.1.0'}`
3. `.github/workflows/dev.yml`：Build frontend 加 `env: VITE_UPDATE_CHANNEL=dev, VITE_APP_VERSION=dev-<SHA>`；publish 加 `Generate dev.json manifest`
4. `.github/workflows/release.yml`：Build frontend 加 `env: VITE_UPDATE_CHANNEL=latest, VITE_APP_VERSION=X.Y.Z`；publish 加 `Generate latest.json manifest`

### JSON 清单格式（updater.ts 的 UpdateManifest 接口）
客户端下载路径：dev 通道 `releases/download/dev-latest/dev.json`；latest 通道 `releases/latest/download/latest.json`

- YAML 语法通过；CI 结构校验通过（env 注入 + JSON 清单步骤到位）

## Bug 修复：设置页 closeAction 开关 + 编辑器保存格式变更

### 1. 设置页 closeAction 开关无响应（stale closure bug）
- **根因**：`FlatGroup` 中 closeAction 单选按钮的 `onClick` 连续调用了两次 `update()`：

- **修复**：在 `onUpdate` 中增加 `if (!transaction.docChanged) return` 检查，跳过所有装饰器 transaction，仅文档内容实际变更时才序列化回存

## 修复 build.cjs 的 cargo PATH 问题

### 背景
用户运行 `node scripts/build.cjs` 时，`cargo build --release` 报错 `'cargo' 不是内部或外部命令`。原因：rustup 安装的 cargo 在 `~/.cargo/bin`，但该目录未加入 cmd.exe/PowerShell 的 PATH（Git Bash 中有，但 `execSync` 在 Windows 上默认用 cmd.exe 执行）。

## 修复新建窗口白屏卡死 + 托盘双进程问题

### 问题1：新建窗口白屏卡死
**根因**：
- `new_window` 命令的 `WebviewWindowBuilder` 缺少 `transparent(true)`，与主窗口配置不一致。Windows + Tauri v2 + WebView2 在透明/不透明窗口混用时，新窗口 WebView2 渲染管线不一致，初始化易白屏/卡死
- `open_devtools` 等命令硬编码 `get_webview_window("main")`，新窗口无法使用
- 新窗口前端重复执行主窗口全部初始化逻辑（含自动更新检查），增加负担

**修复**：
- `lib.rs` `new_window`：添加 `.transparent(true)`，URL 改为 `index.html?win=secondary`
- `lib.rs` `new_window_with_config`：同样添加 `.transparent(true)` + URL 参数
- `lib.rs` `open_devtools`：参数从 `app_handle: AppHandle` 改为 `window: WebviewWindow`，操作当前调用窗口（Tauri 自动注入），新窗口也能打开自己的 DevTools
- `App.tsx`：新增 `isSecondaryWindow`（通过 URL `?win=secondary` 识别），自动更新检查 useEffect 中新窗口跳过

### 问题2：托盘出现两个进程/图标
**根因**（Tauri 官方 Issue #8982）：
- `tauri.conf.json` 的 `app.trayIcon` 配置在 Tauri v2 中**会自动创建一个托盘实例**（无菜单、无事件绑定 → 点击无响应）
- 代码 `lib.rs` setup 中 `TrayIconBuilder::new().build()` 又创建**第二个**托盘实例（有菜单、有事件 → 点击有效）
- 两个托盘叠加：配置创建的无响应 + 代码创建的有效，用户感知为"两个进程"

**修复**：
- `tauri.conf.json`：移除 `app.trayIcon` 配置（消除自动创建的无效托盘）
- `lib.rs` 托盘创建：`TrayIconBuilder::with_id("main-tray")` + 显式 `.icon(app.default_window_icon().unwrap().clone())` + `.tooltip("FkeMark")`（因配置已移除，必须在代码中显式设置图标）
- `lib.rs` single-instance 回调增强：优先恢复 main 窗口（show + unminimize + set_focus），若不存在则聚焦任意可见窗口

### 涉及文件
- `src-tauri/tauri.conf.json`、`src-tauri/src/lib.rs`、`src/App.tsx`

### 校验
- `npx tsc --noEmit`：零错误通过
- `cargo check`：零错误零警告通过

### 关键教训（长期记忆）
1. **Tauri v2 托盘配置陷阱**：`app.trayIcon` 配置会自动创建托盘实例，不能与代码 `TrayIconBuilder` 并存，否则出现两个托盘图标（一个无效）。正确做法：要么纯配置（但无法绑事件），要么纯代码（推荐，可绑菜单/事件 + 显式图标）。
2. **新窗口配置一致性**：`WebviewWindowBuilder` 创建的新窗口必须与主窗口的关键属性（特别是 `transparent`）保持一致，否则 Windows WebView2 渲染异常。
3. **多窗口命令通用性**：涉及窗口操作的 Tauri 命令应接收 `WebviewWindow` 参数（Tauri 自动注入当前窗口），而非硬编码 `get_webview_window("main")`，否则新窗口无法使用。

## 新建窗口白屏卡死根因修复（二次排查）

### 问题
上一轮修复（transparent 一致性 + URL 参数 + 跳过更新检查）后，新建窗口白屏卡死无响应问题**仍然存在**。

### 真正根因（官方 Issue #13092）
Tauri v2 在 **Windows** 上，**同步命令**（`fn`）中调用 `WebviewWindowBuilder::new().build()` 会**死锁 WebView2**，导致新窗口白屏冻结、无法关闭、进程无响应。

官方文档明确警告：
> On Windows, this function deadlocks when used in a synchronous command and event handlers. You should use async commands and separate threads when creating windows.

之前的 `new_window` / `new_window_with_config` 都是同步 `fn`，触发死锁。

### 修复
将两个命令改为 `async fn`：
- `async fn new_window(app_handle)` 
- `async fn new_window_with_config(app_handle, config_path)`

前端 `invoke('new_window')` 调用方式不变（invoke 对 sync/async 命令透明）。

### 涉及文件
- `src-tauri/src/lib.rs`：`new_window`、`new_window_with_config` 改为 async

### 校验
- `cargo check`：零错误零警告通过

### 关键教训（已更新到 MEMORY.md）
**新窗口命令必须 async**——这是 Tauri v2 + Windows 的硬性要求。同步命令创建窗口必然死锁，与 transparent 等配置无关。这是新建窗口白屏卡死的**唯一根因**，之前对 transparent 的调整只是锦上添花。
