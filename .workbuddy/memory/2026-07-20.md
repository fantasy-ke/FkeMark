## 快捷键设置布局/样式主题适配修复

### 问题
用户反馈「快捷键设置中的布局样式不适合主题配置，导致字体不清晰或太明亮」。

### 根因
`src/styles/components.css` 的快捷键区块（`.shortcut-label`/`.shortcut-scope`/`.shortcut-key`/`.shortcut-reset`）误用了主题里**未定义**的 CSS 变量：
- `var(--text, #333)` → 永远走兜底 `#333`（深灰）
- `var(--text-muted, #999)` / `var(--text-muted, #888)` → 永远 `#999`/`#888`
- `var(--bg-elevated, #fafafa)` / `#f0f0f3` → 永远浅色

主题实际只定义了 `--fg`/`--muted`/`--surface`/`--bg`/`--border`/`--accent`(赤陶色 #c96442)/`--accent-soft`/`--accent-foreground`/`--destructive`/`--fg-soft` 等。
后果：暗色主题下深灰字(`#333`)+深色底 → 文字看不清；浅色胶囊(`#fafafa`)+深色面板 → 过亮刺眼；捕获/hover 态还用了写死蓝色 `#4f7cff`，与主题赤陶色不统一。

### 修复
全部改用语义化 token：
- `.shortcut-label` → `color: var(--fg)`（删 `--text`）
- `.shortcut-scope` → `background: var(--fg-soft); color: var(--muted); border: 1px solid var(--border)`（删 `--bg-elevated`/`--text-muted`）
- `.shortcut-key` → `background: var(--bg); color: var(--fg); border: var(--border); border-radius: var(--radius-btn)`，加 `transition`；hover 用 `var(--accent)`
- `.shortcut-key.capturing` → `background: var(--accent-soft); color: var(--accent-foreground)`（删写死蓝色）
- `.shortcut-reset` → `color: var(--muted)`；hover `color: var(--destructive)`
- 布局健壮性：`.shortcut-label` 加 `min-width:0 + ellipsis`，scope/key/reset 加 `flex:none` 防挤压

### 校验
纯 CSS 改动，未改 TS。所有引用 token 均在 `variables.css` 的 light/dark 两主题下定义。

### 遗留（未改，供后续）
`.toast-*` 系列（原 components.css 约 1211–1251 行，现已拆至 `src/styles/components/toast.css`）仍用未定义变量 `--bg-elevated`/`--text`/`--text-muted` 及写死蓝色 `--accent, #4f7cff`，存在同样的明暗适配问题。本次按用户范围仅修快捷键，toast 待单独处理。

## 修复：图片拖拽上传无效果（Tauri v2 事件名变更）

### 问题
用户反馈「拖拽图片进来没有效果，图标也不会上传」。

### 根因（Tauri v1 → v2 事件名变更）
Tauri v2 的文件拖放事件系统全面变更：

| Tauri v1 | Tauri v2 |
|----------|----------|
| `tauri://file-drop` | `tauri://drag-drop`（通过 `onDragDropEvent` API）|
| `tauri://file-drop-hover` | `tauri://drag-over` |
| `tauri://file-drop-cancelled` | `tauri://drag-leave` |
| 新增 | `tauri://drag-enter` |

payload 结构也变了：
- v1: `event.payload` 是 `string[]`（文件路径数组）
- v2: `event.payload` 是 `{ type: 'enter'|'over'|'drop'|'leave', paths: string[], position: PhysicalPosition }`

`App.tsx` 中的代码仍在用 v1 的 `listen('tauri://file-drop', ...)` 监听旧事件名，**该事件在 Tauri v2 中永远不会触发**，导致拖拽文件完全无响应。

### 修复
**`src/App.tsx`**：
- 移除 `import { listen } from '@tauri-apps/api/event'`
- 新增 `import { getCurrentWebview } from '@tauri-apps/api/webview'`
- 移除 `safeTauriListener` 导入（不再使用）
- `useEffect` 中将 `listen('tauri://file-drop', ...)` 替换为 `getCurrentWebview().onDragDropEvent(handler)`
- handler 内检查 `event.payload.type === 'drop'`，使用 `event.payload.paths` 获取文件路径数组
- 清理逻辑改为标准的 `unlisten` + `cancelled` flag 模式

**`src/styles/editor.css`**（图片上传占位节点主题适配）：
- `.fk-img-ph` 背景：`var(--bg-elevated, #f6f6f8)` → `var(--surface)`（未定义变量 → 语义 token）
- `.fk-img-spinner` border-top-color：`var(--accent, #4f7cff)` → `var(--accent)`（移除写死蓝色兜底）
- `.fk-img-name` color：`var(--text, #333)` → `var(--fg)`（未定义变量 → 语义 token）
- `.fk-img-pct` color：`var(--text-muted, #888)` → `var(--muted)`
- `.fk-img-bar-fill` background：`var(--accent, #4f7cff)` → `var(--accent)`
- `.fk-img-cancel` color：`var(--text-muted, #999)` → `var(--muted)`

### 验证
- `tsc --noEmit`：0 错误
- `vite build`：通过（395 modules, CSS 124.80KB, 3.35s）
- lint：0 错误

### 关键教训（Tauri v2 避坑）
**Tauri v2 的文件拖放事件名从 `tauri://file-drop` 改为 `tauri://drag-drop`**。v1 的三个事件被替换为 v2 的四个事件（新增 `drag-enter`），且 payload 从裸 `string[]` 变为带 `type` 字段的联合类型。推荐使用 `getCurrentWebview().onDragDropEvent(handler)` API 而非直接 `listen`，它会统一处理四种事件类型。迁移 v1 → v2 时务必检查所有 `tauri://` 事件名。

## 修复：图片显示失败（Tauri v2 WebView 无法加载相对路径图片）

### 问题
用户反馈「图片拖拽后文件已复制到 assets 目录，但编辑器视图中图片显示失败」。

### 根因（Tauri v2 WebView 的 URL 解析机制）
Tauri v2 的 WebView 从 `http://tauri.localhost/` 加载前端页面。`<img src="./assets/xxx.png">` 会被浏览器按照标准 URL 解析规则解析为 `http://tauri.localhost/assets/xxx.png`——这是一个 HTTP 请求，不是文件系统路径。WebView 只能服务于 `dist` 目录下的静态资源，无法通过相对路径访问用户文件系统上的图片。

### 解决方案
使用 Tauri v2 的 `convertFileSrc()` API + asset 协议，将本地文件路径转为 `http://asset.localhost/...` 协议 URL，WebView 才能正确加载本地图片。

### 修改文件

**1. `src-tauri/tauri.conf.json`**：启用 assetProtocol
```json
"security": {
  "csp": null,
  "assetProtocol": {
    "enable": true,
    "scope": ["**"]
  }
}
```

**2. `src-tauri/Cargo.toml`**：添加 `protocol-asset` feature
```toml
tauri = { version = "2", features = ["tray-icon", "devtools", "macos-private-api", "protocol-asset"] }
```

**3. 新建 `src/utils/asset.ts`**：路径转换工具
- `toAssetUrl(src, docDir)` — 相对路径 `./assets/xxx` → `convertFileSrc(绝对路径)` → `http://asset.localhost/...`
- `toRelPath(src, docDir)` — 逆转换，asset URL → `./assets/xxx`（用于 Markdown 序列化）
- 非 Tauri 环境 / 无 docDir / 非相对路径 / 已是 URL → 原样返回（向后兼容）

**4. `src/utils/markdown.ts`**：接受可选 `docDir` 参数
- `markdownToHtml(md, docDir?)` — 图片 `![](./assets/xxx)` 的 src 经 `toAssetUrl` 转为 asset URL
- `htmlToMarkdown(html, docDir?)` — `<img src="asset_url">` 的 src 经 `toRelPath` 转回相对路径
- `parseInlineMd(text, docDir?)` / `divToMarkdown(el, docDir?)` / `inlineToMd(el, docDir?)` 均透传 docDir
- docDir 为可选参数，不传时行为不变（向后兼容）

**5. `src/components/Editor.tsx`**：传递 docDir + 上传使用 convertFileSrc
- 新增 `docDirRef`（从 `filePath` 提取目录部分）
- 所有 `markdownToHtml` / `htmlToMarkdown` 调用传入 `docDirRef.current`
- `insertImageUploadFromPath`：`upload_asset` 返回 `./assets/xxx` 后经 `toAssetUrl` 转为 asset URL 再设为 img src
- `insertImageUploadFromBlob`：粘贴图片同理

### 数据流（修复后）
```
拖拽图片 → upload_asset 复制文件到 assets/ → 返回 ./assets/xxx
         → toAssetUrl(./assets/xxx, docDir) → http://asset.localhost/C%3A%5C...%5Cassets%5Cxxx
         → <img src="http://asset.localhost/..."> → WebView 正确加载 ✓

保存文件 → htmlToMarkdown(html, docDir) → toRelPath(asset_url, docDir) → ./assets/xxx → ![](./assets/xxx) ✓

打开文件 → markdownToHtml(md, docDir) → toAssetUrl(./assets/xxx, docDir) → <img src="asset_url"> ✓
```

### 验证
- `tsc --noEmit`：0 错误
- `vite build`：通过（391 modules, 3.36s）
- `cargo check`：通过

### 关键教训（Tauri v2 避坑）
**Tauri v2 的 WebView 从 `http://tauri.localhost/` 加载页面，相对路径 `./assets/xxx` 会被解析为 HTTP 请求而非文件系统路径。** 必须使用 `convertFileSrc()` 将本地文件路径转为 `http://asset.localhost/...` 协议 URL。需要三步配置：(1) `tauri.conf.json` 中 `security.assetProtocol.enable = true` + scope；(2) `Cargo.toml` 中 tauri features 加 `protocol-asset`；(3) 前端用 `convertFileSrc()` 转换路径。Markdown 序列化时需逆转换回相对路径以保持文档可移植性。

上一轮修复了 `hide_to_tray` 和 `open_devtools` 的硬编码 "main" 问题（改用 `window` 参数）。用户要求排查其余所有 `get_webview_window("main")` 调用点，防止多窗口场景下出现类似问题。

### 排查结果（共 4 处 `get_webview_window("main")`）

| 位置 | 问题 | 严重度 |
|------|------|--------|
| `show_window` 命令 (L205) | 使用 `app_handle`，若次窗口调用只显示主窗口 | 低（前端从未调用） |
| 单实例插件 (L305) | 已有 fallback 逻辑 | 无 |
| 托盘菜单"显示" (L347) | 无 fallback，主窗口不存在时无效 | 中 |
| 托盘图标点击 (L366) | 同上 + 无法恢复被 `hide_to_tray` 隐藏的次窗口 | **高** |

### 核心风险场景
用户打开新窗口 B → 点关闭选"最小化到托盘" → B 被 `hide_to_tray` 隐藏 → 点托盘图标只恢复主窗口 A → **窗口 B 永久丢失**（隐藏但无法恢复）。

### 修复方案

**1. 新增 `show_app_windows(app: &AppHandle)` 辅助函数**：
- 遍历所有窗口，`show()` + `unminimize()` 恢复每个被隐藏/最小化的窗口
- 优先 `set_focus` 主窗口；主窗口不存在时聚焦最后一个窗口
- 解决次窗口被 `hide_to_tray` 隐藏后无法通过托盘恢复的问题

**2. `show_window` 命令**：`app_handle` → `window` 参数，与 `hide_to_tray` / `open_devtools` 保持一致

**3. 统一调用**：托盘菜单"显示"、托盘图标点击、单实例插件 → 全部改用 `show_app_windows(app)`

### 修改文件
- `src-tauri/src/lib.rs`：
  - `show_window` 签名 `app_handle: AppHandle` → `window: WebviewWindow`
  - 新增 `show_app_windows` 函数（L211-L231）
  - 单实例插件 L325：内联逻辑 → `show_app_windows(app)`
  - 托盘菜单 "show" L356：内联逻辑 → `show_app_windows(app)`
  - 托盘图标点击 L372：内联逻辑 → `show_app_windows(app)`

### 验证
- `cargo check`：通过（exit code 0）
- lint：0 错误
- 前端无需改动（`show_window` 从未被前端调用，签名变更无副作用）

### 关键教训
多窗口应用中，**所有窗口操作命令都应使用 Tauri 自动注入的 `window` 参数**而非 `app_handle.get_webview_window("main")`。系统级事件（托盘点击、单实例）需要恢复所有窗口时，应遍历 `app.webview_windows()` 而非只针对 "main"。

## 修复：Markdown 往返转换仍有 3 个格式丢失问题

### 问题
用户再次检查 `设计文档0.md` 的 git diff，发现上一轮修复后仍存在：
1. 列表标记 `*` 被变更为 `-`
2. 表格分隔行 `| --------- |` 被变更为 `| --- |`
3. 多了很多换行（块级元素之间出现 2 个空行而非 1 个）

### 根因分析

**问题 1（列表标记 `*`→`-`）**：
`markdownToHtml` 在 `<ul>` 上设置了 `data-marker="*"`，但 TipTap 的 `StarterKit` 内置 `BulletList` 扩展不认识这个属性，解析时直接丢弃。`getHTML()` 输出的 `<ul>` 不带 `data-marker`，`listToMd` 读不到就默认用 `-`。

**问题 2（表格分隔行 dash 缩短）**：
`markdownToHtml` 的 `flushTable` 把分隔行存为 `['---']` 占位后丢弃了原始内容。`tableToMd` 只能根据单元格对齐属性重新生成分隔行，始终输出 `---`（3 个 dash）。且 TipTap 的 `Table` 扩展也不保留任何自定义属性。

**问题 3（多余换行）**：
`divToMarkdown` 中每个块级元素前加 `\n`、后加 `\n\n`：
- `<p>` → `\ntext\n\n`
- `<h1>` → `\n# title\n\n`
- 两个相邻块：`\ntext\n\n` + `\n# title\n\n` = `\ntext\n\n\n# title\n\n`
- trim 后：`text\n\n\n# title` → 3 个换行 = 2 个空行
- 原始 Markdown 只有 2 个换行 = 1 个空行

### 修复方案

**Fix 1：自定义 BulletList 扩展保留 marker 属性**

`src/components/Editor.tsx`：
- 新增 `import BulletList from '@tiptap/extension-bullet-list'`
- 创建 `CustomBulletList = BulletList.extend({ addAttributes() { ... marker ... } })`
  - `marker` 属性：`parseHTML` 读取 `data-marker`，`renderHTML` 输出 `data-marker`（仅非默认值 `-` 时输出）
- `StarterKit.configure` 中禁用 `bulletList: false`
- extensions 数组中用 `CustomBulletList` 替代默认 BulletList

**Fix 2：自定义 Table 扩展保留 separators 属性**

`src/components/Editor.tsx`：
- 创建 `CustomTable = Table.extend({ addAttributes() { ... separators ... } })`
  - `separators` 属性：`parseHTML` 读取 `data-separators`，`renderHTML` 输出 `data-separators`
- extensions 数组中用 `CustomTable.configure(...)` 替代 `Table.configure(...)`

`src/utils/markdown.ts`：
- `markdownToHtml` 的 `flushTable`：把 `tableBuffer[1]`（原始分隔行）join 后存入 `<table data-separators="...">`
- `markdownToHtml` 主循环：分隔行不再存为 `['---']` 占位，改为 `parseTableRow(trimmed)` 保留原始内容
- `tableToMd`：优先读取 `data-separators` 属性，有则用原始分隔行（补齐列数），无则根据对齐生成

**Fix 3：归一化连续空行**

`src/utils/markdown.ts`：
- `htmlToMarkdown` 返回值添加 `.replace(/\n{3,}/g, '\n\n')`
- 3+ 连续换行 → 2 个换行（1 个空行），与 `markdownToHtml` 的归一化逻辑对称

### 修改文件
- `src/components/Editor.tsx`：+39 行（CustomBulletList + CustomTable 扩展定义 + extensions 配置）
- `src/utils/markdown.ts`：+24/-10 行（flushTable 存 separators + tableToMd 读 separators + htmlToMarkdown 归一化换行）

### 验证
- `tsc --noEmit`：0 错误
- `vite build`：通过（404 modules, 3.61s）
- lint：0 错误
- `设计文档0.md`：已 `git checkout` 恢复到原始版本

### 关键教训
TipTap 的默认扩展不保留自定义 HTML 属性。要让 MD→HTML→TipTap→HTML→MD 往返转换无损，必须对每个需要保留属性的扩展使用 `.extend({ addAttributes() })` 模式：
1. `parseHTML`：从 HTML 元素读取 data-* 属性
2. `renderHTML`：在条件满足时输出 data-* 属性
3. 在 `StarterKit.configure` 中禁用默认扩展，用自定义扩展替代

这与之前 `StyledOrderedList`（`data-ls` 属性）的模式完全一致。
