# 2026-07-19 工作日志

## 新窗口 splash 启动画面闪烁修复

### 问题描述
新建窗口时会短暂出现"初始化界面"（splash 启动画面：FM 图标 + "FkeMark" 标题 + loading spinner），闪一下后才进入实际编辑器界面。期望：新建窗口应直接显示实际界面，不出现闪烁。

### 根因
`index.html` 的 `#root` 内预置了 splash 启动画面（静态 HTML）：
```html
<div id="root">
  <div class="splash-screen">
    <div class="splash-icon">FM</div>
    <h1 class="splash-title">FkeMark</h1>
    <div class="splash-loading"></div>
  </div>
</div>
```

新窗口加载流程：
1. WebView2 加载 index.html → **splash 立即显示**（静态 HTML，无需 JS）
2. JS 加载、React 挂载（`ReactDOM.createRoot(#root).render(<App />)`）→ **覆盖 #root 内容**，splash 被替换为实际界面

从窗口创建到 React 挂载完成的时间窗口内（几十到几百毫秒），splash 短暂可见——这就是"闪一下"。

主窗口启动时 splash 是预期的加载画面（用户预期看到应用启动加载），但新窗口是用户主动触发的，预期立即看到编辑器，先看到 splash 就显得突兀。

### 修复方案
**新窗口延迟显示**：窗口创建时 `visible(false)`，前端 React 渲染完成后再调用 `window.show()`。这样 splash 在窗口不可见时就已经被替换，用户看到的就是最终界面。

### 修改清单

| 文件 | 改动 |
|---|---|
| `src-tauri/src/lib.rs` | `new_window`：`.center()` 后添加 `.visible(false)` |
| `src-tauri/src/lib.rs` | `new_window_with_config`：`.center()` 后添加 `.visible(false)` |
| `src/App.tsx` | 导入 `getCurrentWebviewWindow`（from `@tauri-apps/api/webviewWindow`） |
| `src/App.tsx` | `showOnboarding` 初始状态：secondary 窗口强制 `false`（避免首启引导闪现） |
| `src/App.tsx` | `loadSettings` 的 useEffect：`.finally()` 后，secondary 窗口用**双 RAF** 调用 `getCurrentWebviewWindow().show()` 显示窗口 |

### 关键代码
```typescript
// App.tsx — loadSettings 完成后显示新窗口
useEffect(() => {
  loadSettings().finally(() => {
    if (!isSecondaryWindow || !isTauri()) return
    // 双 RAF：确保 React 完成首屏渲染并绘制后再显示窗口
    requestAnimationFrame(() => {
      requestAnimationFrame(() => {
        getCurrentWebviewWindow()
          .show()
          .catch((e) => console.error('Failed to show secondary window:', e))
      })
    })
  })
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [])
```

### 踩坑点
- `getCurrentWebviewWindow()` 在 Tauri v2 中**同步返回 `WebviewWindow` 实例**（非 Promise），不能 `.then()`，直接 `.show().catch(...)` 调用。`show()` 方法返回 Promise。
- 双 RAF（嵌套 `requestAnimationFrame`）确保至少经过一次浏览器绘制，React 重渲染完成。单 RAF 可能不够。
- 即使 `loadSettings` 失败（invoke 报错），`.finally()` 仍会执行 show，保证窗口不会永远不可见。

### 校验
- `npx tsc --noEmit`：零错误通过
- `cargo check`：零错误零警告通过（2.28s）

### 测试建议
`npm run tauri dev` 后点击"新建窗口"：
1. 新窗口应**直接显示编辑器界面**，不再先闪现 splash 启动画面
2. 新窗口不应显示首启引导（Onboarding）
3. 新窗口显示略有延迟（React 渲染完成才显示），但无闪烁，体验更佳

## macOS 打包修复：`transparent` 方法不存在

### 问题
macOS CI 打包（`tauri build --target x86_64-apple-darwin --bundles dmg`）报错：
```
error[E0599]: no method named `transparent` found for struct `WebviewWindowBuilder<'a, R, M>`
  --> src/lib.rs:163:6  (new_window)
  --> src/lib.rs:206:6  (new_window_with_config)
```
Windows 上 `cargo check` 通过，但 macOS 编译失败。

### 根因
`WebviewWindowBuilder::transparent()` 方法的平台标注为：
```rust
#[cfg(any(not(target_os = "macos"), feature = "macos-private-api"))]
pub fn transparent(mut self, transparent: bool) -> Self
```
即 **Windows/Linux 始终可用，但 macOS 上只有启用 `tauri` 的 `macos-private-api` feature 时该方法才被编译进来**。macOS 透明窗口需调用私有 API，Tauri 默认不暴露该方法。之前为修复 Windows 白屏给新窗口加了 `.transparent(true)`（Windows 可用），但 macOS 编译时该 method 不存在 → 报错。

### 修复（两步，保持 macOS 新窗口与主窗口 `transparent: true` 配置一致）
1. `src-tauri/Cargo.toml`：`tauri` 依赖追加 `macos-private-api` feature：
   ```toml
   tauri = { version = "2", features = ["tray-icon", "devtools", "macos-private-api"] }
   ```
   （非 macOS 平台该 feature 为空操作，不影响 Windows/Linux 构建）
2. `src-tauri/tauri.conf.json`：`app` 下加 `"macOSPrivateApi": true`：
   ```json
   "app": { "withGlobalTauri": true, "macOSPrivateApi": true, "windows": [...] }
   ```
   否则 `tauri-build` 的 feature allowlist 校验会报 "Cargo.toml features does not match allowlist defined under tauri.conf.json"。

### 校验
- Windows `cargo check`：零错误通过（allowlist 校验也过）
- macOS 编译需 CI 验证（本地无法跑 macOS target）；逻辑已确认：`macos-private-api` 启用后 `transparent` 方法在 macOS 存在

### 注意
- 启用 `macos-private-api` 会让 macOS 构建依赖私有 API，**无法上架 Mac App Store**（GitHub 分发 dmg 不受影响）——若未来要上架需改用条件编译排除 macOS 的 `transparent` 调用（但会失去 macOS 新窗口透明）。
