# FkeMark 产品设计与系统策划方案

## 一、 产品定位与核心理念

* **产品名称**：FkeMark
* **一句话定义**：一款无数据库、文件系统优先、高颜值的极简 Markdown 混合即时渲染编辑器。
* **核心理念**：
* **文件即数据**：软件不建立任何私有数据库或索引。双击 `.md` 文件即开即写，关闭即走。
* **极简视觉**：全面承袭 Tolaria 的扁平化（Flat UI）、无缝窗口（Borderless）以及优雅的现代冷色调排版。
* **所见即所得**：光标所在处显示 Markdown 源码，光标移开即刻渲染为精美排版，杜绝传统分屏编辑的割裂感。

## 三、 系统架构设计

FkeMark 采用 **Tauri + React + TypeScript + ProseMirror** 的技术栈，其前后端职责与数据流向设计如下：

```
+---------------------------------------------------------------------------------+
|                                 前端 WebView (React)                             |
|                                                                                 |
|   +-----------------------+   User Typing   +-------------------------------+   |
|   |  TipTap / ProseMirror | --------------> |   TyporaActiveSyntaxPlugin    |   |
|   |       编辑画布        | <-------------- |   (动态计算光标位置与 Decorator) |   |
|   +-----------------------+                 +-------------------------------+   |
|               ^                                             |                   |
|       Render  |                                             | Debounce Save     |
|               |                                             v                   |
|   +-----------------------+                 +-------------------------------+   |
|   |  TOC 大纲 / 侧边文件树 |                 |      序列化为纯 Markdown 文本    |   |
|   +-----------------------+                 +-------------------------------+   |
+-------------------------------------------------------------|-------------------+
                                                              | IPC Call
                                                              v
+---------------------------------------------------------------------------------+
|                                 Rust 核心 (Tauri)                               |
|                                                                                 |
|   +-----------------------+                 +-------------------------------+   |
|   |   双击关联/命令行启动  |                 |     异步非阻塞 File System I/O |   |
|   |   (提取 OS 传入的文件路径) |             |     (安全写入到本地物理磁盘)     |   |
|   +-----------------------+                 +-------------------------------+   |
+---------------------------------------------------------------------------------+

```

### 1. 前端渲染层 (WebView)

* **富文本状态维护**：基于 ProseMirror 的 Schema，将整个文档看作一个单一的连续文本流，不使用 BlockNote 的“硬性行块”设计。
* **实时转译**：利用 `@tiptap/markdown` 扩展，实时在内存中将 Markdown 字符转换为 ProseMirror 语法树（AST）。

### 2. 后端控制层 (Tauri/Rust)

* **零数据库设计**：彻底放弃任何本地数据库（如 SQLite）或复杂索引（如 Vector DB），所有配置以本地 JSON 文件形式存在用户的 `AppData`/`Application Support` 中。
* **文件系统直连**：利用 Rust 的 `tokio::fs` 进行文件异步读取与写入，防止编辑大文件时界面出现卡顿。

## 四、 核心编辑逻辑与 WYSIWYG 实现机制

要实现 Typora 级别的即时渲染，其核心编辑逻辑不是“实时将 Markdown 字符串渲染成 HTML”，而是“在富文本 DOM 树中，根据光标动态插入/隐藏 Markdown 符号的装饰器（Decoration）”。

### 1. 光标追踪与符号显隐逻辑

* **初始化状态**：当用户输入 `## 这是一个二级标题` 后，ProseMirror 会将其解析为 `heading (level=2)` 节点。默认在界面上，前面的 `## ` 标记是不占位、不显示的。
* **聚焦（Focus）阶段**：
当用户点击或将光标移动到该标题内部时，插件监听到光标 Selection 变化：
1. 计算当前光标在 AST 中的 Node 位置。
2. 如果处于 `heading` 节点，使用 `Decoration.widget` 在该行的头部动态插入一个包含 `## ` 字符的虚拟 `span`（带有 `.md-delimiter` 类名）。
3. 如果在粗体 `文字` 内部，在粗体内容前后分别插入 `` 虚拟字符。


* **失焦（Blur）阶段**：
当光标离开当前行或当前 Mark 标记范围时，重新触发 Plugin 的 `apply` 方法。清空该节点上的所有 Widget Decoration，界面瞬间变回干净的排版状态。

### 2. 连续排版防抖技术

为了避免频繁插入和删除 DOM 节点导致光标跳动（Layout Jitter），我们在 CSS 中对 `.md-delimiter` 进行了过渡动画设计（使用 `max-width` 结合 `opacity`，而不是直接 `display: none`），使其在 150 毫秒内平滑淡入淡出，获得极为舒适的视觉流动感。

## 七、 系统性能指标 (System Performance Metrics)

作为一款本地编辑器，极致的性能是它抗衡 Notion 等云端笔记的核心武器。FkeMark 必须满足以下性能硬指标：

1. **冷启动时间**：双击本地任意 `.md` 文件至界面完全渲染完成，耗时控制在 **400 毫秒** 以内。
2. **内存占用**：在没有任何文档打开的情况下，常驻内存控制在 **60MB - 80MB** 之间（得益于 Tauri 抛弃了 Chromium 内核，直接调用系统原生 WebView）。
3. **大文件支持**：加载一个大小为 **10MB**（约合数百万字）的巨型 Markdown 文档，解析并首次渲染时间不超过 **1.5秒**，且在连续快速输入时**无任何感知延迟（输入延迟小于 16毫秒/单帧）**。
4. **无损数据安全**：在断电或强制杀进程测试中，由于采用了基于 Debounce 300 毫秒的高频自动保存，用户最多只会丢失最近一次停笔后不超过 **1秒** 的输入内容。
